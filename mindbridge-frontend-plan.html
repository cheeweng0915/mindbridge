Product Goal

What MindBridge is

A student mental health support web app that helps students do a quick stress check-in, understand their current burnout level, and move toward helpful actions such as support resources, journaling, or a breathing exercise.

What the frontend must achieve first

Make the full user journey feel complete and trustworthy even before the backend exists. The UI should be clear, calm, supportive, and demo-ready using sample data, React state, and browser storage.

Color and Tone Direction

Primary palette

  • Primary: teal-500 `#14b8a6`
  • Primary light: teal-50 `#f0fdfa`
  • Secondary: violet-400 `#a78bfa`
  • Background: slate-50 `#f8fafc`

Text and surfaces

  • Text: slate-800 `#1e293b`
  • Muted text: slate-500 `#64748b`
  • Cards: white `#ffffff`
  • Use rounded corners, soft shadows, generous whitespace.

Stress colors

  • Low: emerald-400 `#34d399`
  • Medium: amber-400 `#fbbf24`
  • High: rose-400 `#fb7185`
  • Use supportive wording with these colors, not alarmist tone.

Page-by-Page Frontend Features and Functions

1. Home Page

What the user can do: understand the app, start the quiz, open resources, read the safety disclaimer.

  • Hero headline with supportive tone
  • Main CTA: Start Check-In
  • Secondary CTA: Explore Resources
  • Feature cards for Quiz, Journal, Breathing, Resources
  • Short disclaimer: support tool, not diagnosis

Frontend function: build trust fast and route users into the main flow.

2. Quiz Page

What the user can do: answer 8 to 10 questions, move between questions, see progress, submit.

  • Progress bar
  • Question card or carousel
  • Answer pills like Never / Sometimes / Often
  • Back and next controls
  • Submit button after final question

Frontend function: collect answers in state, calculate score, save result to localStorage, then route to `/result`.

3. Result Page

What the user can do: view stress level, read suggestions, move to helpful next actions.

  • Color-coded stress result card
  • Short explanation of the result
  • Suggestion list based on level
  • Action buttons: Breathing, Journal, Resources, Retake Quiz
  • Fallback if no stored result: go back to quiz

Frontend function: read stored result safely on mount and render the correct suggestion set.

4. Resources Page

What the user can do: find crisis help, self-help options, campus support, and trusted contacts.

  • Emergency banner pinned near the top
  • Grouped resource categories
  • Cards with title, type, short description, action label
  • Clear urgent vs non-urgent separation

Frontend function: provide a safe, fast-scanning help path without needing live backend data.

5. Journal Page

What the user can do: choose mood, write reflection, save entry, review older entries.

  • Emoji mood picker
  • Short text entry area
  • Save button
  • History list from localStorage
  • Optional empty-state card for first visit

Frontend function: CRUD-like local entry management using browser storage only.

6. Breathing Page

What the user can do: start, pause, reset, and follow a breathing rhythm visually.

  • Animated inhale / hold / exhale circle
  • Phase label
  • Cycle counter
  • Timer display
  • Start, pause, reset controls

Frontend function: pure client-side timing and animation using state plus CSS transitions.

UI Example: Screen-by-Screen Flow

This section explains the UI in a more visual way: what the user sees first, which sections exist on the page, what they can click, and what happens after the click.

Screen 1: Home Page

User sees first: a calm hero section with the app name, a short message like "Take a gentle mental check-in," and two main buttons.

Main sections on screen:

  • Hero section
  • Main CTA buttons
  • Feature cards for Quiz, Resources, Journal, and Breathing
  • Short disclaimer at the bottom

User can click:

  • `Start Check-In` -> go to `/quiz`
  • `Explore Resources` -> go to `/resources`
  • Feature cards -> go to their related page

Purpose of this page: make the app feel safe, friendly, and easy to start.

Screen 2: Quiz Page

User sees first: a progress bar at the top, then one big centered question card.

Main sections on screen:

  • Progress indicator
  • Question text
  • Answer buttons like Never / Sometimes / Often
  • Back and Next buttons

User can click:

  • An answer option -> selected state appears
  • `Next` -> next question appears
  • `Back` -> previous question appears
  • `Submit` on the final question -> score is calculated and saved

After final submit: navigate to `/result`.

Screen 3: Result Page

User sees first: a large result card with the stress level, a calm explanation, and supportive next-step buttons.

Main sections on screen:

  • Stress level card: Low / Medium / High
  • Short explanation text
  • Suggestion list
  • Action buttons

User can click:

  • `Try Breathing Exercise` -> go to `/breathing`
  • `Open Resources` -> go to `/resources`
  • `Write Journal Entry` -> go to `/journal`
  • `Retake Quiz` -> go back to `/quiz`

Purpose of this page: give emotional feedback and guide the user to the right next action.

Screen 4: Resources Page

User sees first: an emergency help banner at the top, then grouped resource cards below.

Main sections on screen:

  • Emergency / crisis banner
  • Urgent help resources
  • Campus or student support resources
  • Self-help and wellness resources

User can click:

  • Resource card -> open link, call action, or detail page
  • Back navigation -> return to previous page

Purpose of this page: provide the practical help path, especially after a high-stress result.

Screen 5: Journal Page

User sees first: a mood picker row, a text area for reflection, and saved entries underneath.

Main sections on screen:

  • Emoji mood picker
  • Prompt text such as "How are you feeling today?"
  • Journal text area
  • `Save Entry` button
  • Mood history cards

User can click:

  • An emoji -> mood is selected
  • `Save Entry` -> entry is stored and shown in history

After click: the new entry appears below without needing a backend.

Screen 6: Breathing Page

User sees first: a large breathing circle centered on screen, a phase label, and simple controls underneath.

Main sections on screen:

  • Animated breathing circle
  • Phase label: Inhale / Hold / Exhale
  • Countdown or timer
  • Cycle counter
  • Start / Pause / Reset buttons

User can click:

  • `Start` -> animation and timer begin
  • `Pause` -> current cycle pauses
  • `Reset` -> return to the starting state

Purpose of this page: give the user one immediate calming action after the quiz.

Main User Journey Example

  1. User lands on Home and clicks `Start Check-In`.
  2. Quiz page opens and shows Question 1 plus progress bar.
  3. User answers all questions and clicks `Submit`.
  4. Result page opens and shows stress level plus suggestions.
  5. User chooses one next step:
    • if they want calm support now -> click `Try Breathing Exercise`
    • if they want practical help -> click `Open Resources`
    • if they want to reflect -> click `Write Journal Entry`
  6. Later, user revisits the app and still sees saved journal or quiz data from localStorage where relevant.

Client-Side Data Flow

Quiz to Result

  1. Quiz page stores answer state in React state.
  2. Submit action runs score calculation from `quiz-scoring.ts`.
  3. Save result object to `localStorage`.
  4. Navigate to `/result` without relying on URL params.
  5. Result page loads saved result in `useEffect`.
  6. If result is missing, route back to `/quiz`.

Journal and Breathing

  • Journal entries stored under a key like `mindbridge-journal`.
  • All storage reads happen inside `useEffect` to avoid hydration mismatch.
  • Breathing timer uses local component state plus `setInterval`.
  • Resource content stays in constants for the local demo.

Technical Notes

  • Tailwind CSS 4 theme setup belongs in `globals.css`, not `tailwind.config.js`.
  • All `localStorage` reads should happen in `useEffect` for hydration safety.
  • Use pure CSS transitions and React state before reaching for heavy libraries.
  • Keep `Home`, `Resources`, and layout polished first so the app feels complete early.
  • For the first version, hardcoded resources and suggestion lists are acceptable.

Verification Checklist

  1. All 6 routes load without errors.
  2. Quiz flow runs end-to-end and stores the result.
  3. Result page shows correct level and suggestions.
  4. Journal entry persists after refresh.
  5. Breathing timer starts, pauses, resets, and counts cycles correctly.
  6. App looks usable at mobile width around 375px.
  7. `npm run build` completes without TypeScript errors.

Short Recommendation

If time gets tight, ship this order first: Home, Quiz, Result, Resources. Those four pages already create a complete, competition-ready user journey. Add Journal and Breathing after the core flow is stable.

This planning file is intentionally frontend-focused so the UI can be built and demoed with local browser storage.
